# 数据源稳定性方案总结

## 目的

在不进入正式开发的前提下，明确本项目关于数据源稳定性的总体设计原则、主备策略、失败处理方式和结果透明机制。

一句话结论：

> 数据源稳定性不是附加优化，而是整个量化系统可信度的基座。

---

## 1. 核心原则

本项目不追求“任何时候都一定抓得到所有数据”，而是追求：

```text
1. 某个数据源失败时，系统不至于整体崩溃
2. 某类数据缺失时，系统知道是否可以降级运行
3. 抓到数据后，系统能够判断数据是否真的可用
4. 每日评分和推送结果必须明确标识数据可信度
5. 数据异常时，系统不能输出看似正常但实际错误的结果
```

也就是说，稳定性的目标不是：

```text
100% 抓取成功
```

而是：

```text
失败可识别
异常可追踪
结果可解释
系统可降级
历史可回补
```

---

## 2. 各类数据源主备策略

### 2.1 行情数据

```text
主源：AkShare
备源：Tushare / 东方财富
```

说明：

```text
- 行情数据是全系统最核心的数据类型
- 日 K / 基础行情必须稳定
- 如果主源失败，应优先切换到备源
```

---

### 2.2 公告数据

```text
主源：巨潮资讯
备源：交易所公告 / 东方财富公告
```

说明：

```text
- 公告是资讯因子的重要来源
- 允许延迟，但不能长期缺失
- 公告失败不应直接拖垮整套评分链路
```

---

### 2.3 新闻数据

```text
主源：东方财富 / 财联社
备源：证券时报 / 中国证券报 / 上海证券报
```

说明：

```text
- 新闻主要服务于解释和资讯评分
- 可以部分失败
- 但失败时要明确告知用户“解释能力下降”
```

---

### 2.4 财务数据

```text
主源：Tushare
备源：东方财富 / AkShare
```

说明：

```text
- 财务数据重要，但更新频率较低
- 允许使用最近一期有效快照兜底
- 不应因财务数据当天抓不到而导致系统崩溃
```

---

### 2.5 板块与概念数据

```text
主源：东方财富概念板块
备源：同花顺概念 / AkShare
```

说明：

```text
- 板块概念数据主要服务于市场状态和题材拥挤度判断
- 可以影响评级和仓位约束
- 如果缺失，应降低环境判断可信度
```

---

## 3. 数据优先级分层

不是所有数据都必须以同样标准稳定可得。

### 3.1 一级核心数据，必须尽量稳定

```text
- 日 K / 基础行情
- 股票基础信息
- 市场整体行情
```

影响：

```text
这些数据缺失时，系统不应继续正式评分和正式推送。
```

---

### 3.2 二级重要数据，允许降级

```text
- 公告
- 新闻
- 财务数据
- 板块概念数据
```

影响：

```text
这些数据缺失时，系统可以降级运行，但必须标注结果可信度下降。
```

---

### 3.3 三级辅助数据，非核心依赖

```text
- 资金流向
- 龙虎榜
- 分钟级数据
- 情绪参考社区数据
```

影响：

```text
这些数据在 MVP 中只作为辅助，不应成为核心买卖触发依据。
```

---

## 4. 数据源分层设计

建议增加统一的数据源抽象层：

```text
data_collectors/
├── providers/
│   ├── akshare_provider.py
│   ├── tushare_provider.py
│   ├── eastmoney_provider.py
│   ├── cninfo_provider.py
│   └── base_provider.py
├── quote_collector.py
├── news_collector.py
├── fundamental_collector.py
└── data_quality_checker.py
```

核心思想：

```text
业务层不关心数据来自哪里
业务层只关心数据是否完整、是否可信
```

例如：

```text
quote_collector 先调 AkShare
失败则调 Tushare
再失败则调东方财富
仍失败则记录日志并标记当日行情不完整
```

---

## 5. 必须有数据质量校验

不是“抓到了就能用”，而是“抓到后还要校验”。

### 5.1 行情数据校验

每次入库前至少检查：

```text
open / high / low / close 不能为 0
high >= open / close / low
low <= open / close / high
成交额不能为负
涨跌幅不能明显异常
交易日必须正确
股票数量不能明显少于正常水平
```

例如：

```text
正常可交易股票约 5000 只
今天只抓到 800 只
不能视为正常完成
```

应直接标记：

```text
行情数据不完整，不生成正式评分
```

---

### 5.2 财务数据校验

至少检查：

```text
营收、利润、ROE 等字段允许为空
但不能出现明显异常脏值
例如：
- ROE = 9999%
- 资产负债率 = -300%
```

若财务抓取失败：

```text
- 不应导致系统崩溃
- 可使用最近一期财务快照
- 或将基本面分降为中性分 / 降低权重
```

---

### 5.3 资讯数据校验

至少检查：

```text
- 标题是否为空
- 发布时间是否存在
- 正文是否明显缺失
- 当日数据量是否异常偏低
- 股票代码识别是否大量失败
```

若资讯数据异常：

```text
- 不应阻塞行情评分
- 但应降低资讯分可信度
- 并在日报中提示解释能力下降
```

---

## 6. 失败重试与切源机制

每个数据任务都应该具备重试能力。

建议机制：

```text
第一次失败：等待 10 秒重试
第二次失败：等待 1 分钟重试
第三次失败：切换备用源
第四次失败：记录失败，不阻塞整套系统
```

### 6.1 行情抓取失败

```text
- 重试 3 次
- 尝试备用源
- 若仍失败，标记今日行情状态 = incomplete
- 停止正式评分和正式日报
```

### 6.2 新闻抓取失败

```text
- 重试 2 次
- 若仍失败，则跳过新闻模块
- 不影响纯行情评分
- 日报中明确说明新闻模块降级
```

### 6.3 财务抓取失败

```text
- 使用最近一期财务数据
- 标记 source_status = stale
- 基本面分析继续，但可信度下降
```

---

## 7. 缓存与历史数据兜底

不要每天完全依赖在线接口。

### 7.1 行情数据兜底

每日成功抓取后入库。

如果某天接口失败：

```text
可以使用历史数据库继续计算：
- MA5
- MA10
- MA20
- MA60
```

但如果以下关键数据缺失：

```text
- 今日收盘价
- 今日涨跌幅
- 今日成交额
```

则：

```text
不能生成正式日报
不能生成正式评分结果
```

---

### 7.2 财务数据兜底

财务数据不是每天变化，可以缓存最近一期。

如果今天抓取失败：

```text
- 使用最近一期财务数据
- 标记 source_status = stale
```

---

### 7.3 股票基础信息兜底

股票基础信息可每日或每周更新。

如果当天更新失败：

```text
- 可沿用昨日股票池
- 但 ST 状态、停牌状态最好尽量每日校验
```

---

## 8. 每日结果必须输出“数据可信度”

系统每天不只输出股票评分，还应输出：

```text
data_confidence = 高 / 中 / 低
```

建议示例规则：

| 情况 | 数据可信度 |
|---|---|
| 行情完整率 > 98%，财务正常，资讯正常 | 高 |
| 行情完整率 > 95%，资讯失败 | 中 |
| 行情完整率 < 90% | 低 |
| 主源和备源都失败 | 低 |
| 今日交易日判断异常 | 低 |

如果数据可信度低，推送中必须明确提示：

```text
【数据风险提醒】
今日行情数据完整率仅 82%，本日评分仅供参考，不建议根据结果开仓。
```

---

## 9. 飞书推送必须显示数据状态

日报不能只推结果，还要推结果对应的数据质量。

建议在日报中加入：

```text
【数据源状态】
行情数据：成功，完整率 99.2%
财务数据：成功，使用最近一期财报
公告数据：成功，新增 126 条
新闻数据：部分失败，东方财富超时，已使用备用源
LLM 结构化：成功 82 条，失败 4 条
数据可信度：中
```

这样用户不会误以为系统“正常工作”，却不知道数据实际上有缺口。

---

## 10. MVP 阶段必须做到的稳定性能力

数据源稳定性应纳入 MVP，不是后期再补的增强项。

MVP 至少要做到：

```text
1. 每类核心数据定义主源
2. 关键数据支持一个备源或可切换方案
3. 抓取后做基础质量校验
4. 记录抓取日志与错误状态
5. 区分硬失败和软失败
6. 支持降级运行
7. 推送前检查数据完整性
8. 支持按日期回补历史缺口
9. 输出每日数据可信度
10. 在飞书日报中透明展示数据状态
```

---

## 11. 这套方案的本质

这套方案的目标不是让系统显得“更高级”，而是保证：

```text
- 策略验证不会因为数据异常而失真
- 收益、回撤、胜率等指标具备可信度
- 用户能分清“策略失效”还是“数据异常”
- 系统失败时不会输出误导性结果
```

一句话总结：

> 用主备数据源保证可抓，用质量校验保证可用，用重试和缓存保证可持续，用可信度和状态展示保证结果不误导人。
